# ZenixUI — Product Principles

This document captures how we think about product decisions — not engineering decisions. When a feature is proposed, evaluate it here first, before evaluating the architecture.

---

## The Ten Principles

### 1. Customer buys an Experience

Not components. Not blocks. Not tokens. Not a design system.

The customer opens ZenixUI and sees "SaaS Startup" or "Business Landing." They evaluate it as a complete website. They customize it as a complete website. They receive it as a complete website.

The fact that it is internally composed from Hero, Features, Pricing, and Footer sections is our implementation detail. It is invisible to them, and it should remain invisible.

### 2. Studio is the product

Without the Studio, ZenixUI is a template marketplace. There are thousands of those. The Studio is what transforms a static download into a configured, personalized frontend.

The Studio does not need to be powerful. In v1, it only needs to do five things well:
- Change the logo and brand name
- Change the color palette
- Change the fonts
- Edit key text (headline, CTA, pricing)
- Show an accurate live preview

That is sufficient for the product promise.

### 3. Export is ownership

Export is not a download. It is a transfer of ownership. The moment a customer clicks Export, the project belongs entirely to them. They owe us nothing. They depend on us for nothing. They can take the project to any developer, any agency, any platform.

This is our moat. Other tools create lock-in. We create trust.

### 4. Generated code must look handwritten

A senior frontend engineer reviewing the exported project should not be able to identify it as generated. The code must look like it was written by an experienced developer following best practices.

This means:
- Semantic HTML elements, not `<div>` soup
- Meaningful component and variable names
- Consistent formatting
- No `// Generated by ZenixUI` comments
- No AI-style verbose inline comments
- No redundant abstractions that a human wouldn't have written

### 5. Zero vendor lock-in

The exported project must contain zero `@zenix/*` dependencies. No proprietary runtime. No required service. No hidden dependency on zenixui.com.

This is a non-negotiable promise. Breaking it — even once, even for convenience — breaks the product.

### 6. Earn every abstraction

Every package, folder, hook, or shared utility must justify its existence through real, observable duplication that already exists. We do not build for anticipated future needs.

When a new abstraction is proposed, ask: "What would break or duplicate if we didn't create this?" If the answer is "nothing yet," wait.

### 7. Simplicity over cleverness

A simple solution that a junior developer can understand beats a clever solution that requires tribal knowledge. Cleverness makes code harder to hand off, harder to modify, and harder to debug.

When two solutions produce the same outcome, always choose the simpler one.

### 8. The exported project must outlive ZenixUI

Imagine zenixui.com shuts down tomorrow. Every customer's exported project continues to work perfectly, forever. This is the test. If this scenario reveals a dependency, the dependency must be removed.

### 9. Every feature must have a customer

Before building anything, state who benefits and how. Use specific customer types:
- Solo founder who needs a website before a pitch
- Freelancer delivering a project to a client by Friday
- Agency that builds similar sites repeatedly and wants to go faster

If the beneficiary is "the architecture" or "future engineers" — it is infrastructure. Delay it until the product demands it.

### 10. Ship before expanding

One complete vertical slice delivered beautifully is worth more than five partial slices. v1 is one experience, end-to-end, working perfectly. Nothing else ships until that is true.

## Studio Design Philosophy

The Studio must feel like Canva, not like a React configuration panel.

### Canva-style labels (use these)

| Section | Label |
|---|---|
| hero.headline | Headline |
| hero.subheadline | Supporting Text |
| hero.primaryCta | Primary Button |
| theme.primaryColor | Brand Color |
| theme.backgroundColor | Background |
| theme.radius | Corner Style |

### Implementation-style labels (never use these)

- `headline` → React prop name, not a user label
- `primaryColor` → CSS variable name, not a user label
- `radius: "md"` → Enum value, not a user choice

The Studio UI translates between the user's mental model ("Brand Color") and the implementation ("--brand-primary"). The user never sees the implementation side.

## Marketing Philosophy

We do not lead with the mechanism. We lead with the benefit.

| ❌ Mechanism-first | ✅ Benefit-first |
|---|---|
| "Export clean Next.js code" | "Own your frontend completely" |
| "Customize with CSS variables" | "Change colors and fonts instantly" |
| "Composed from reusable sections" | "Launch in hours, not weeks" |

**The headline on the homepage should answer:** "What do I get?"

Not: "How does it work?"

### Our best lines

- "Choose a launch-ready frontend. Customize it visually. Own the code."
- "Export code you'll forget was generated."
- "So natural, it feels handwritten."
- "Launch your startup website in hours."

The last point explains what it is. The first three explain why it matters.

---

## Decision Framework

When a product or engineering decision is unclear, apply this sequence:

1. **Does the customer benefit?** If yes, proceed. If no, stop.
2. **Does it conflict with any of the ten principles?** If yes, revise or reject.
3. **Does it belong in v1?** If no, add to the Icebox in `roadmap.md`.
4. **Does it earn its abstraction?** If no, implement locally first.
5. **Can you explain it without implementation language?** If no, simplify the concept.
